# Zeabur Deployment — Implementation Plan
# Manhole Cover Design Analysis Pipeline

## Architecture Overview

```
GitHub repo
    │
    └─► Zeabur (auto-deploy on push)
            │
            ├── Python service (Flask dashboard + auth + pipeline scripts)
            │       app.py                   ← web entrypoint + auth + DATA_DIR setup
            │       01_scrape_data.py        ← text + image scraping
            │       03_sentiment_analysis.py ← RoBERTa NLP sentiment
            │       04_image_processing.py   ← VLM + LLM image analysis
            │       05_cross_analysis.py     ← sentiment × image correlation
            │       06_confusion_matrix.py   ← human vs AI validation
            │       06_visualize_images.py   ← image grid visualizations
            │       07_evaluate_design.py    ← design opportunity scoring
            │       08_generate_report.py    ← final report generation
            │
            ├── Persistent Volume → /data/   (path controlled by DATA_DIR env var)
            │       /data/images/            ← downloaded images (~500MB)
            │       /data/hf_cache/          ← HuggingFace model cache
            │       /data/text_raw.csv
            │       /data/image_metadata.csv
            │       /data/text_with_sentiment.csv
            │       /data/grade_sample.csv
            │       /data/human_grades.csv
            │       /data/output/            ← charts + CSVs for download
            │
            └── Environment Variables (set in Zeabur dashboard)
                    DATA_DIR                ← storage root (default: /data)
                    ACCESS_KEY              ← base32 key for dashboard login
                    SESSION_SECRET          ← Flask session signing secret
                    OPENROUTER_API_KEY      ← for VLM+LLM image analysis
                    YOUTUBE_API_KEY         ← for extra scraping coverage (optional)
                    ENABLE_MASTODON         ← toggle Mastodon enrichment (default true)
                    MASTODON_INSTANCES      ← comma-separated Mastodon instances
                    ENABLE_PINTEREST        ← toggle Pinterest enrichment (default false)
                    ENABLE_MAPILLARY        ← toggle Mapillary enrichment (default false)
                    MAPILLARY_ACCESS_TOKEN  ← required only when Mapillary is enabled
```

## Step 0 — Generate Auth Keys (do this first, locally)

```bash
# Access key — share this with teammates who need dashboard access
python -c "import base64, os; print(base64.b32encode(os.urandom(10)).decode())"
# Example output: MFRA2YTBMJQXIZLB

# Session secret — keep this private, never share or commit
python -c "import secrets; print(secrets.token_hex(32))"
# Example output: 3f2a9c1d...
```

Save both outputs — you'll need them in Step 4.

## Step 1 — Prepare the GitHub Repo

Structure your repo exactly like this at the root:

```
gettingaids2/
├── app.py                      ← must be at root (Zeabur entrypoint)
├── requirements.txt            ← must be at root
├── zbpack.json                 ← must be at root
├── runtime.txt                 ← must be at root
├── 01_scrape_data.py
├── 03_sentiment_analysis.py
├── 04_image_processing.py
├── 05_cross_analysis.py
├── 06_confusion_matrix.py
├── 06_visualize_images.py
├── 07_evaluate_design.py
└── 08_generate_report.py
```

Add a .gitignore:

```
data/
output/
*.csv
*.png
hf_cache/
__pycache__/
.env
```

IMPORTANT — DATA_DIR path handling (already configured in current code):
  • All scripts read DATA_DIR = os.getenv('DATA_DIR', '/data')
  • On Zeabur: set env var DATA_DIR=/data (matches the persistent volume mount)
  • Locally: scripts default to ./data if DATA_DIR is not set
  • app.py prints [DIAG] lines at startup showing the resolved DATA_DIR
  • app.py falls back to ./data if /data is not writable
  • TRANSFORMERS_CACHE is set dynamically inside 03_sentiment_analysis.py

## Step 2 — Create Zeabur Project

1. Go to zeabur.com → Dashboard → New Project
2. Add Service → Git → connect GitHub → select your repo
3. Zeabur auto-detects Python from requirements.txt
4. zbpack.json handles build + start command automatically

---

## Step 3 — Mount Persistent Volume

CRITICAL — without this, all scraped data wipes on every redeploy.

Zeabur dashboard → your service → Volumes tab:
  Volume ID:  pipeline-data
  Mount Path: /data

Do this before the first run. Cannot be added retroactively without data loss.

## Step 4 — Set Environment Variables

Zeabur dashboard → your service → Variables tab.

### Required variables

  DATA_DIR               = /data                 ← must match volume mount path
  ACCESS_KEY             = MFRA2YTBMJQXIZLB     ← from Step 0
  SESSION_SECRET         = 3f2a9c1d...           ← from Step 0

### Optional feature variables

  ADMIN_RESET_PASSWORD   = your_admin_password   ← enables admin reset controls on dashboard
  OPENROUTER_API_KEY     = your_openrouter_key   ← enables AI image analysis (Stage 3)
  YOUTUBE_API_KEY        = your_youtube_key      ← enables YouTube scraping (Stage 1)
  ENABLE_MASTODON        = true                  ← Stage 1 Mastodon text/image enrichment
  MASTODON_INSTANCES     = mastodon.social,mastodon.world,fosstodon.org
  ENABLE_PINTEREST       = false                 ← Stage 1 Pinterest enrichment (best-effort)
  ENABLE_MAPILLARY       = false                 ← Stage 1 Mapillary enrichment (token-gated)
  MAPILLARY_ACCESS_TOKEN = MLY|...               ← required only if ENABLE_MAPILLARY=true

Never put these in code or commit them to GitHub.

### How to set the Admin Reset Password

  Choose any strong password you'll remember. This unlocks the admin panel
  on the dashboard (soft reset, reset analysis, full reset, reset & rerun).

  Example:
    ADMIN_RESET_PASSWORD = MyStr0ngP@ssw0rd!

  Without this variable, the admin reset buttons on the dashboard will not work.

### How to get a YouTube API Key

  1. Go to https://console.cloud.google.com/
  2. Create a new project (or select an existing one)
  3. Navigate to APIs & Services → Library
  4. Search for "YouTube Data API v3" → click Enable
  5. Go to APIs & Services → Credentials
  6. Click "Create Credentials" → "API Key"
  7. (Recommended) Restrict the key to "YouTube Data API v3" only
  8. Copy the key and paste it into Zeabur Variables as YOUTUBE_API_KEY

  Without this key, Stage 1 still runs — it just skips YouTube enrichment
  and you'll see in logs: "YOUTUBE: skipped (no YOUTUBE_API_KEY)"

### How to get an OpenRouter API Key

  1. Go to https://openrouter.ai/
  2. Sign up / log in
  3. Go to Keys → Create new key
  4. Copy the key and paste it into Zeabur Variables as OPENROUTER_API_KEY

  Without this key, Stage 3 runs in metadata-only mode (no AI visual analysis).

### Stage 1 source toggles (Mastodon / Pinterest / Mapillary)

  - Mastodon is enabled by default in code. To disable:
      ENABLE_MASTODON = false

  - Pinterest is disabled by default (recommended unless needed, as it is fragile):
      ENABLE_PINTEREST = true

  - Mapillary is disabled by default and requires a token:
      ENABLE_MAPILLARY = true
      MAPILLARY_ACCESS_TOKEN = MLY|your_token

  - Optional Mastodon instance override:
      MASTODON_INSTANCES = mastodon.social,mastodon.world,fosstodon.org

  Notes:
  - Pinterest scraping is best-effort and may degrade if upstream anti-bot behavior changes.
  - Mapillary is useful for geo street imagery but may include non-manhole scenes.

## Step 5 — Set Resources

Zeabur dashboard → your service → Resources:
  RAM:  2 GB minimum  (RoBERTa inference needs ~1.5GB)
  CPU:  0.5 vCPU      (sufficient for CPU-only inference)

## Step 6 — Deploy

Push to GitHub → Zeabur auto-builds.

First build: ~8–12 minutes (torch is ~800MB, downloads once and caches).
Subsequent deploys: ~2–3 minutes (cached layers).

Access your dashboard at: https://your-service.zeabur.app

---

## Step 7 — Run the Pipeline

1. Open your Zeabur URL
2. Enter your ACCESS_KEY at the login screen
3. Select "Run Full Pipeline" or individual stages
4. Log streams live, auto-refreshes every 5 seconds
5. Download outputs from the Outputs section when done

Full pipeline runtime estimate: 2–4 hours on Zeabur shared CPU.

## Security Summary

| Threat                  | Protection                                       |
|-------------------------|--------------------------------------------------|
| Unauthorized access     | Session-based auth, key required on every login  |
| Brute force login       | Rate limited: 10/min, 30/hour per IP             |
| API key exposure        | Env vars only, never in code or HTTP responses   |
| Source code exposure    | Not possible — Zeabur serves container not code  |
| Session hijacking       | SESSION_SECRET signs cookies server-side         |

## Cost Management

Zeabur is pay-as-you-go. To minimize cost:
  - Suspend the service after each pipeline run
  - Resume only when you need to rerun or download outputs
  - Suspended services cost ~$0

Rough estimate if running only during active use (~10 hrs total):
  Compute (2GB RAM):   ~$0.50–1.00
  Persistent volume:   ~$0.10/month
  Total for project:   < $2

## Gotchas

1. TORCH BUILD TIMEOUT
   If Zeabur times out installing torch, add to requirements.txt:
     --extra-index-url https://download.pytorch.org/whl/cpu
   And pin: torch==2.1.0+cpu

2. GUNICORN TIMEOUT
   zbpack.json sets --timeout 300 for slow pipeline triggers.
   Do not lower this.

3. SINGLE WORKER
   gunicorn runs 1 worker to prevent duplicate pipeline runs.
   Do not increase workers.

4. VOLUME IS THE ONLY PERSISTENT STORAGE
   Anything written outside /data/ is wiped on restart/redeploy.
   All scripts use DATA_DIR env var (default /data) to locate the volume.

5. SCRAPING SKIPS YOUTUBE / AI IMAGE ANALYSIS
   If you see "YOUTUBE: skipped" in Stage 1 logs, set YOUTUBE_API_KEY.
   If Stage 3 says "OPENROUTER_API_KEY not set", set OPENROUTER_API_KEY.
   Both are optional — the pipeline still runs without them, just with less data.

6. OPTIONAL SOURCE FLAGS (MASTODON / PINTEREST / MAPILLARY)
   If logs show these sources are skipped, check:
   a) ENABLE_MASTODON / ENABLE_PINTEREST / ENABLE_MAPILLARY values
   b) MAPILLARY_ACCESS_TOKEN present when ENABLE_MAPILLARY=true
   c) MASTODON_INSTANCES list is valid and reachable from Zeabur runtime

   Expected skip logs include:
     "[PINTEREST] skipped (ENABLE_PINTEREST=false)"
     "[MAPILLARY] skipped (ENABLE_MAPILLARY=false)"
     "[MAPILLARY] skipped (ENABLE_MAPILLARY=true but MAPILLARY_ACCESS_TOKEN is missing)"

7. "PIPELINE RUNS BUT /data IS EMPTY"
   This is the #1 deployment issue. Causes in order of likelihood:
   a) Persistent volume not mounted → check Volumes tab, Mount Path must be /data
   b) DATA_DIR env var not set or wrong value → check it equals /data
   c) Volume mounted but app lacks write permission → check startup logs for [DIAG] lines
   d) Build cache serving stale code → clear build cache in Zeabur and redeploy
   To verify: check Zeabur deploy logs for lines like:
     [DIAG] DATA_DIR = /data
     [DIAG] /data writable = True

8. RATE LIMIT STATE IS IN-MEMORY
   flask-limiter uses memory storage (resets on service restart).
   Fine for a student project — not suitable for high-security production.
